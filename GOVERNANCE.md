# Governance

## Governance goals

This project is governed to maximize:

- scientific credibility
- software quality
- reproducibility
- contributor clarity
- maintainable growth

## Roles

### Steering maintainers

Responsible for:

- roadmap decisions
- release approval
- policy changes
- conflict resolution
- design-partner relationships

### Domain leads

Suggested domains:

- behavioral methods
- EEG/MEG
- Bayesian / cognitive modeling
- open science / reproducibility
- research software engineering

Domain leads review changes touching their area before a feature is marked “supported”.

### Advisory board

Small external advisory group that provides periodic review on:

- standards alignment
- community relevance
- benchmark design
- publication strategy

## Decision process

### Routine decisions

Handled by the relevant maintainer or domain lead.

### Architecture or policy changes

Require:

1. an issue or discussion
2. a short ADR when the change is material
3. approval from at least two maintainers, including one domain lead where relevant

### Release decisions

A release candidate may be cut only when:

- CI is green
- docs are synchronized
- examples run
- supported skills meet their release gates
- security-sensitive changes have been reviewed

## Authorship and credit

The project values both scientific and software contributions.

Track credit for:

- code
- standards design
- documentation
- benchmark curation
- design-partner feedback
- review and maintenance

Use `CITATION.cff`, release notes, and contributor metadata consistently.

## Design-partner program

Before broad feature expansion, recruit a small number of labs that agree to:

- test end-to-end workflows
- provide structured feedback
- report friction points
- validate domain relevance

## Community process

Use:

- Issues for actionable work
- Discussions for design questions and community Q&A
- pull requests for reviewed changes
- release notes for public communication
